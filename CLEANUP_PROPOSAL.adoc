== Context

The `cloudera-partners` repo currently contains 5 git submodules (`ClouderaSetup`, `DataServicesLabs`, `End2EndLabs`, `PartnerResources`, `PSEAutomation`), a top-level `README.adoc`, CI workflows, and two untracked local directories (`Testing/`, `utilities/`) that have accumulated assorted scripts and Terraform artifacts. Before pushing anything upstream, we'd like to tidy this up and add a dedicated area for cluster testing (on-prem and cloud).

== Proposed Cleanup

. *Delete OS junk* — remove `Testing/._os_metrics.json` (macOS AppleDouble artifact).
. *Stop tracking large binaries* — relocate the 153 MB and 191 KB `.tar` files out of the repo (or gitignore). These should not live in git history.
. *Resolve `utilities/terraform/`* — orphaned state without source. Propose deleting (`terraform.tfstate` in particular should never be committed). If the `.tf` source lives elsewhere, pull it in before we delete.
. *Expand `.gitignore`* — add `.terraform/`, `*.tfstate*`, `*.tar`, `._*`, `*.log` (current `.gitignore` only covers `.DS_Store`, `.vscode/`, `.idea/`).
. *Introduce a cluster-testing directory* — new top-level `ClusterTests/` (or similar) with `OnPrem/` and `Cloud/` subdirectories, mirroring the `ClouderaSetup/OnPremise` naming convention.
. *Relocate the existing `Testing/` scripts* — fold the benchmark scripts into `ClusterTests/Benchmarks/` so we don't have two parallel "test" concepts.

== Open Questions

. *Replace or coexist?* Should the new cluster-testing directory *replace* `Testing/`, or live alongside it? Preference here is to fold benchmarks in and drop `Testing/`.
. *Scope of "testing"* — which of the following should the new dir support?
+
* [ ] Smoke / functional validation of a running cluster
* [ ] Infrastructure provisioning tests (Terraform / Ansible)
* [ ] Performance benchmarks (what's currently in `Testing/`)
* [ ] All of the above
. *`utilities/terraform/`* — OK to delete? Or does someone have the `.tf` source elsewhere that should be restored first?
. *Untracked binaries* — OK to delete the `.tar` files and `._os_metrics.json` outright, or should they be archived somewhere (e.g. S3, internal share) first?
. *Directory name* — `ClusterTests/`, `ClusterTesting/`, `ClusterValidation/`, or something else?
. *Cloud scope* — which providers should the `Cloud/` subdirectory cover initially? AWS, Azure, GCP, all three?
